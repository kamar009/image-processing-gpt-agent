# Спецификация API: тип `furniture_portfolio`

Документ фиксирует **контракт и продуктовые правила** для сценария «фото мебели после монтажа → публикация в портфолио (сайт / соцсети)». Реализация в коде — шаги A2+; настоящий файл — единый источник правды до начала разработки.

См. также общее описание интеграции: [INTEGRATION.md](INTEGRATION.md).

---

## 1. Назначение

- **`type=furniture_portfolio`**: исходник — фото **установленной офисной мебели** на объекте (часто с телефона), подготовка к публикации на **сайте** и в **соцсетях**.
- **Пользователь:** менеджер в офисе.
- **Объём v1:** одиночная загрузка (`POST /process-image`), без пакетной очереди в этом документе.

---

## 2. Поля запроса `POST /process-image` (multipart/form-data)

| Поле | Обязательность | Описание |
|------|----------------|----------|
| `image` | Да | Файл изображения (JPEG, PNG, WebP — как в общем API). |
| `type` | Да | Для этого сценария: **`furniture_portfolio`**. |
| `furniture_scene` | Да, если `type=furniture_portfolio` | Подтип помещения; enum, см. [§3](#3-значения-furniture_scene). |
| `output_target` | Да, если `type=furniture_portfolio` | Целевой пресет вывода; enum, см. [§4](#4-значения-output_target-и-целевые-размеры). |
| `enhanced` | Нет | Усиленная **программная** обработка v1 (шумоподавление, резкость и смежные шаги в пайплайне; в `operations` появляется `furniture_enhanced_software_v1`). Вкл: **`1`**, **`true`**, **`on`** (регистр не важен); иначе — выкл. Управляется **только** этим полем запроса. Людей из кадра **не** удаляет. |

**Прочие поля** (`background`, `format`, `max_output_kb`, `vision_provider`, `vision_model`, `style`, `crop_mode`, `quality_level`, …): действуют правила общего API. Для **v1** после реализации следует явно задокументировать в коде/README: какие поля **переопределяются** пресетом `output_target`, какие **игнорируются** для `furniture_portfolio`. До реализации — **TBD**.

## 3. Значения `furniture_scene`

| Значение API | Отображение (RU) |
|--------------|------------------|
| `executive_office` | Кабинет руководителя |
| `meeting_room` | Переговорная комната |
| `reception_waiting` | Ресепшен и зона ожидания |
| `open_workspace` | Рабочие места пользователей |
| `retail_counters` | Торговое оборудование (буфеты, прилавки) |
| `archive_library` | Архивы и библиотеки |
| `lounge` | Комната отдыха |

Любое иное значение при `type=furniture_portfolio` → ошибка валидации (ожидается **422** с понятным `detail`).

## 4. Значения `output_target` и целевые размеры

Целевые **ширина × высота** и **ориентировочный лимит размера выходного файла** (для настройки пресета и валидации):

| Значение | Назначение | Выход (px) | Лимит выхода (ориентир) |
|----------|------------|------------|-------------------------|
| `site` | Портфолио на сайте | 1600 × 900 | 400 КБ |
| `banner` | Баннер на сайте | 1920 × 900 | 450 КБ |
| `social_vk` | ВКонтакте | 1080 × 1080 | 350 КБ |
| `social_telegram` | Telegram (канал, 16:9) | 1280 × 720 | 300 КБ |
| `social_max` | MAX (вертикаль под ленту) | 1080 × 1350 | 350 КБ |

Любое иное значение при `type=furniture_portfolio` → **422** с понятным `detail`.

## 5. Продуктовые правила (не код)

- **Фон:** реальный, как на объекте; не замена на вымышленный интерьер.
- **Запрещено искажать изделие:** нельзя менять **цвет ЛДСП**, **фурнитуру**, **форму модулей** (геометрия гарнитура).
- **Кроп:** допустим, в том числе если часть мебели уходит из кадра.
- **Отражения (стекло, глянец):** цель — убрать **постороннее** в отражениях; без режима `enhanced` возможности ограничены программным пайплайном (честно отражать в UX/доках).
- **Люди в кадре:** **предупреждение** в ответе (не обязательный отказ). Режим `enhanced` (v1) **не** удаляет людей; при `enhanced` + люди в кадре добавляется отдельное предупреждение в `validation_warnings`. Отдельный этап retouch / удаление людей — при будущей реализации.
- **Минимальное разрешение входа:** для `type=furniture_portfolio` длинная сторона **≥ 1200 px** после учёта EXIF-ориентации; детали — [§8](#8-отказы-и-http), [§9](#9-параметры-входа-a4).

---

## 6. Пример запроса (curl)

```bash
curl -sS -X POST "https://api.example.com/process-image" \
  -F "image=@./photo.jpg" \
  -F "type=furniture_portfolio" \
  -F "furniture_scene=meeting_room" \
  -F "output_target=site" \
  -F "enhanced=0"
```

## 7. Пример успешного ответа (фрагмент JSON)

Поля ниже — **дополнения** к уже существующему телу ответа `POST /process-image` (см. [INTEGRATION.md](INTEGRATION.md): `file_id`, `download_url`, `validation_*`, метрики и т.д.).

| Поле | Описание |
|------|----------|
| `type` | `furniture_portfolio` |
| `furniture_scene` | Эхо запроса |
| `output_target` | Эхо запроса |
| `enhanced_requested` | Булево — запрошено ли усиление (`enhanced` в форме) |
| `enhanced_applied` | Булево — совпадает с `enhanced_requested` при успешном ответе (усиление v1 по флагу в запросе) |
| `people_detected` | Булево — оценка Vision: в кадре видны люди |
| `operations` | Список операций пайплайна; при усилении v1 включает `furniture_enhanced_software_v1` (и связанные шаги) |
| `validation_warnings` | Массив строк: люди в кадре; при `enhanced` + люди — доп. текст, что автоудаления нет |

## 8. Отказы и HTTP

| Ситуация | HTTP | Тело |
|----------|------|------|
| Неверный `furniture_scene` или `output_target` | **422** | `detail` (человекочитаемо), при наличии `request_id` |
| Ниже минимального разрешения входа (`furniture_portfolio`) | **422** | `detail`: фактическая длинная сторона после EXIF и требуемый минимум (см. [§9](#9-параметры-входа-a4)) |
| Прочие ошибки | Как в [INTEGRATION.md](INTEGRATION.md) | `detail`, заголовок `x-request-id` где применимо |

## 9. Параметры входа (A4)

1. **Минимальное разрешение входа** для `type=furniture_portfolio`: **длинная сторона ≥ 1200 px** после применения EXIF Orientation (`ImageOps.exif_transpose` / эквивалент). Меньшая сторона не ограничивается отдельно. Константа в коде: `FURNITURE_PORTFOLIO_MIN_INPUT_LONG_SIDE_PX` (= 1200). Для остальных `type` этот порог **не** применяется.
