# Решение по поставщику Vision (после пилота)

Документ для фиксации выбора провайдера анализа изображений (Sber / OpenAI / Yandex / др.) по результатам пилота и ограничениям бизнеса.

## 1. Критерии

| Критерий | Что измерить | Порог / заметки |
|----------|----------------|-----------------|
| Качество | Субъективная оценка кадров по сценариям (`product`, `banner`, `furniture_portfolio`, …) | Минимальный уровень, согласованный с контентом/маркетингом |
| Стоимость | Цена за запрос / токены, месячный потолок | Бюджет и лимиты договора |
| Задержка p95 | Время ответа Vision из `/metrics` или логов (`vision_ms`) | SLO продукта (например &lt; N с для UX) |
| Надёжность | Доля fallback / 5xx, квоты | Допустимый % деградации |
| Соответствие 152-ФЗ / хранение данных | Где обрабатывается изображение | Требования комплаенса |

## 2. Входные данные пилота

- Зафиксировать набор тестовых изображений и одинаковый протокол запуска для всех провайдеров.
- Сохранить сводку: таблица «файл × провайдер × время × итог (ok/fallback) × заметка».
- Промежуточные CSV/отчёты можно хранить **вне репозитория** (см. [VISION_PILOT_REPO_POLICY.md](VISION_PILOT_REPO_POLICY.md)); в git — только методология и итоговое решение. Локальная структура тестовых кадров: [test_datasets/vision_pilot/README.md](../test_datasets/vision_pilot/README.md).

## 3. Итоговая фиксация (заполняет ответственный)

- **Выбранный провайдер по умолчанию для прода:** …
- **Переменные окружения:** `VISION_PROVIDER`, модель (если применимо), ключи в `.env` (не в git).

- **Запасной вариант:** например `vision_provider=fallback` или второй API при недоступности основного.

## 4. Связь с кодом и деплоем

- Дефолты для SWEB: см. `deploy/sweb/docker-compose.yml` (`VISION_PROVIDER`, `SBER_VISION_MODEL`); переопределение через `PROD_VISION_PROVIDER` / `PROD_SBER_VISION_MODEL` в `.env` на сервере.
- После смены провайдера: smoke `POST /process-image` по ключевым `type` и мониторинг расхода квот.
