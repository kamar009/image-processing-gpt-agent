# Этап 2: SpaceWeb + Docker Compose (только публичный сценарий)

Документ для **руководителя и исполнителя**: вынести сервис в интернет на VPS **SpaceWeb** по уже готовым файлам репозитория, чтобы работала цепочка **загрузка картинки → обработка → скачивание результата**. Внутренний режим (Telegram, очередь задач, Mini App) **на этом этапе не обязателен** — см. раздел «Границы этапа».

Технические детали деплоя и SSL: [SWEB_RUNBOOK.md](SWEB_RUNBOOK.md). Контракт API: [INTEGRATION.md](INTEGRATION.md).

---

## Цель этапа 2 (что считаем успехом)

1. По адресу вида `https://api.ваш-домен` открывается сервис (в т.ч. страница документации API или веб-форма, если включены).
2. Запрос **`GET /health`** возвращает успех (код **200**, в теле есть признак работоспособности, например `"status":"ok"`).
3. Запрос **`POST /process-image`** с реальным файлом возвращает **200** и JSON с полями **`file_id`**, **`download_url`**.
4. По **`download_url`** или **`GET /outputs/{file_id}`** файл **скачивается** (не 404).

Если все четыре пункта выполнены — **этап 2 закрыт** с точки зрения продукта «загрузил — получил файл».

---

## Границы этапа (что сознательно не делаем сейчас)

- Не требуем настройки Telegram, whitelist, **`POST /internal/jobs`**, воркера очереди как бизнес-процесса.
- Не считаем обязательной проверку **`GET /internal/health`** в режиме внутреннего MVP (она нужна, когда включён internal-режим и настроены бот и БД).
- Детальный hardening сервера выходит за рамки этой инструкции — базовый порядок уже в [SWEB_RUNBOOK.md](SWEB_RUNBOOK.md).

## Что нужно заранее (чеклист до начала)

| № | Требование | Зачем |
|---|------------|--------|
| 1 | VPS SpaceWeb с Ubuntu 22.04/24.04, доступ по SSH | Как в runbook; локально: пользователь **`deploy`**, ключ **`~/.ssh/sweb_deploy_github2`** — см. **§2b** в [SWEB_RUNBOOK.md](SWEB_RUNBOOK.md) |
| 2 | DNS: запись типа **A** для поддомена API (например `api.example.com`) на **публичный IP** VPS | HTTPS и стабильный `PUBLIC_BASE_URL` |
| 3 | Код проекта в **`/opt/app`** (клон репозитория или деплой из CI) | Единый путь в документации |
| 4 | Ключ для выбранного vision-провайдера (для текущего прод-дефолта Sber: **`SBER_VISION_AUTH_KEY`** или `SBER_VISION_API_KEY`) | Этап анализа изображения в пайплайне |
| 5 | Решение по лимиту загрузки: **`MAX_UPLOAD_MB`** в `.env` | Согласовать с ожидаемым размером фото |

**Контроль:** есть IP сервера, логин SSH, поддомен «указывает» на этот IP (проверка: `ping api.ваш-домен` или онлайн-проверка DNS).

## Шаг 1. Подготовка сервера (bootstrap)

**Действие (под пользователем с sudo, один раз):**

```bash
sudo APP_USER=deploy APP_DIR=/opt/app bash deploy/sweb/bootstrap-vps.sh
```

**Комментарий:** скрипт создаёт пользователя `deploy`, базовые каталоги и зависимости под Docker. Это «фундамент», без него дальше по инструкции репозитория не поедем.

**Контроль:** можно зайти под `deploy`: `sudo -iu deploy`, перейти в `cd /opt/app` и видеть файлы проекта.

Если под **root** при `git pull` появляется сообщение про **dubious ownership**, один раз: `git config --global --add safe.directory /opt/app` (как в [SWEB_RUNBOOK.md](SWEB_RUNBOOK.md)).

---

## Шаг 2. Файл окружения `.env` (без секретов в Git)

**Действие (пользователь `deploy`, каталог `/opt/app`):**

```bash
cd /opt/app
bash deploy/sweb/init-env-from-example.sh
nano /opt/app/.env
```

**Для этапа 2 (публичный API)** в `.env` должны быть как минимум:

- Для текущего прод-дефолта на SWEB: **`SBER_VISION_AUTH_KEY`** (предпочтительно) или `SBER_VISION_API_KEY`.
- **`INTERNAL_MODE=0`** — публичный сценарий без обязательной настройки Telegram и internal-эндпоинтов.
- **`PUBLIC_BASE_URL=https://api.ваш-домен`** — **без** завершающего `/`. Нужен для **абсолютных** ссылок **`download_url`** в ответе JSON. Если пропустить — клиент может получить относительный путь и «сломать» скачивание вне браузера.
- При необходимости: **`MAX_UPLOAD_MB`**, **`OUTPUT_DIR`** в контейнере задаётся compose; локально в `.env` смотрите [`.env.example`](../.env.example).

Примечание по провайдеру для SWEB: в `deploy/sweb/docker-compose.yml` зафиксирован прод-дефолт
`VISION_PROVIDER=sber` и `SBER_VISION_MODEL=GigaChat-2-Max` (локальные дефолты проекта не меняются).

**Комментарий:** секреты только на сервере в `/opt/app/.env`, права на файл ограничены (`chmod 600` делает скрипт инициализации). В репозиторий ничего не коммитим.

**Контроль:** файл `/opt/app/.env` существует, в нём заполнены `OPENAI_API_KEY` и `PUBLIC_BASE_URL` с **реальным** HTTPS-адресом API.

## Шаг 3. Сборка и запуск контейнеров

**Действие:**

```bash
cd /opt/app
docker compose -f deploy/sweb/docker-compose.yml build
docker compose -f deploy/sweb/docker-compose.yml up -d
```

**Комментарий:** поднимаются reverse-proxy (nginx), API и вспомогательные сервисы из [deploy/sweb/docker-compose.yml](../deploy/sweb/docker-compose.yml). До выпуска SSL-сертификата контейнер **nginx** может жаловаться на отсутствие сертификатов — это ожидаемо; смок API изнутри см. ниже.

**Контроль:**

```bash
docker compose -f deploy/sweb/docker-compose.yml exec -T api curl -fsS http://127.0.0.1:8000/health
```

Ожидается **успешный ответ** (не ошибка curl). Если команда падает — смотреть логи: `docker compose -f deploy/sweb/docker-compose.yml logs api --tail 100`.

## Шаг 4. HTTPS (Let’s Encrypt)

**Действие:** по [SWEB_RUNBOOK.md](SWEB_RUNBOOK.md), типично:

```bash
bash deploy/sweb/issue-cert.sh api.ваш-домен ваш-email@домен
```

(замените хост и email на свои).

**Комментарий:** после появления сертификатов nginx отдаёт **HTTPS** на 443.

**Контроль:** с любой машины в интернете:

```bash
curl -fsS https://api.ваш-домен/health
```

Ожидается **200** и JSON с признаком «ок».

## Шаг 5. Приёмка «загрузка → обработка → скачивание»

### 5.1 Браузер (самый наглядный контроль для руководителя)

1. Откройте **`https://api.ваш-домен/docs`** — должна открыться страница Swagger.
2. Либо откройте **`https://api.ваш-домен/`** — редирект на веб-форму загрузки (если она развёрнута в образе).

**Контроль:** страница открывается без ошибки сертификата (после шага 4).

### 5.2 Один осознанный запрос обработки

Исполнитель с машины, где есть `curl` и тестовое фото:

```bash
curl -sS -X POST "https://api.ваш-домен/process-image" \
  -F "image=@/path/to/photo.jpg" \
  -F "type=product" \
  -F "background=keep" \
  -F "format=webp"
```

**Контроль:** в ответе JSON есть **`file_id`**, **`download_url`** начинается с **`https://api.ваш-домен/...`** (если `PUBLIC_BASE_URL` задан верно).

Скачивание:

```bash
curl -fsS -o /tmp/out.webp "https://api.ваш-домен/outputs/ПОДСТАВИТЬ_file_id"
```

(расширение может быть `.webp`/`.jpg` — ориентируйтесь на ответ API; при 404 см. логи и точный `file_id`.)

### 5.3 Скрипт из репозитория (для автоматизации)

На машине с клоном проекта и Python:

```bash
python scripts/integration_smoke.py --image path/to/file.png --base-url https://api.ваш-домен --strict
```

**Контроль:** скрипт завершается с кодом **0**.

## Типичные сбои (коротко «что сказать разработчику»)

| Симптом | Возможная причина |
|---------|-------------------|
| `download_url` без домена | Не задан или неверен **`PUBLIC_BASE_URL`** |
| 413 | Файл больше **`MAX_UPLOAD_MB`** или лимит прокси |
| 401/502 вокруг OpenAI | Ключ, квота или сеть до OpenAI |
| 404 на `/outputs/...` | Неверный `file_id`, очистка каталога, или запрос до записи файла |
| nginx в `Restarting`, в логах `APP_DOMAIN` / `cannot load certificate` | На сервере старый `nginx.conf` с TLS-плейсхолдером без файлов LE: обновите репо и перезапустите nginx или выполните `issue-cert.sh`; в актуальном репозитории по умолчанию сначала **HTTP на 80** |
| до выпуска сертификата нет HTTPS | Норма: проверяйте `http://ваш-домен/health` на порту **80**, затем `issue-cert.sh` |
