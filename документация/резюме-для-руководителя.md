## О чём проект

Компания получает **сервис подготовки изображений для сайта**. Пользователь (или внутренний сотрудник через Telegram) загружает фото; система **автоматически** приводит его к нужному формату под разные задачи сайта: товар, категория, баннер, интерьер портфолио, а также сценарий **«мебель на объекте → портфолио / соцсети»** (`furniture_portfolio` — отдельный контракт полей и размеров). При необходимости подключается **искусственный интеллект (анализ изображения)** через выбранного провайдера, чтобы лучше понять кадр и корректно обрезать фон, композицию и качество. На выходе — **готовый файл** (например WebP) с контролем качества и ограничений по размеру и времени обработки.

**Два режима использования:**

- **Публичный / классический** — «загрузил → сразу получил результат».
- **Внутренний** — через **Telegram Mini App**: очередь задач, история, доступ только для разрешённых пользователей.

Проект рассчитан на **развёртывание на своём сервере** (в т.ч. у хостинга), с документацией по запуску и проверкам в репозитории (`docs/`).

## Зачем это бизнесу

| Эффект | Описание |
|--------|----------|
| Единый стандарт | Одинаковые правила формата, веса и вида картинок на сайте. |
| Меньше ручной работы | Меньше рутины у дизайнеров и контента на подготовку файлов. |
| Контроль качества | Система может отклонить заведомо плохой или неподходящий результат. |
| Удобный внутренний канал | Telegram снижает порог входа для сотрудников без отдельного сложного ПО. |

## Как устроено в целом (без деталей реализации)

Сервис состоит из **четырёх функциональных модулей** (подробно — в файле [модули-системы.md](модули-системы.md)):

1. **Движок обработки** — «что делаем с картинкой».
2. **Публичный API** — «один запрос — один результат» для интеграций и простой формы.
3. **Внутренняя платформа** — очередь, доступ через Telegram, учёт задач.
4. **Клиентские интерфейсы** — веб-форма и Mini App в Telegram.

**Клиент** обращается либо к **быстрому каналу (модуль 2)**, либо к **внутренней очереди (модуль 3)**; фактическая обработка изображения выполняется **движком (модуль 1)**. **Модуль 3** отвечает за то, **кто** имеет право работать и **в каком порядке** выполняются задачи.

## Риски и ограничения

- **ИИ-провайдер** — качество, скорость и стоимость зависят от внешнего API и лимитов; нужен действующий ключ и мониторинг расходов.
- **Внутренний режим** — требует корректной настройки доступа и секретов на сервере; это задача эксплуатации, не только «включить в интерфейсе».
- **Публичный доступ** к обработке нужно осознанно ограничивать политикой компании, иначе сервисом могут воспользоваться посторонние.

## Этап 2: внешний сервер (SpaceWeb), простой вариант

**Задача:** вынести сервис в интернет так, чтобы сотрудник или интеграция могли **загрузить фото**, получить обработанный файл и **скачать** его по ссылке. Telegram и «внутренняя очередь» на этом шаге **не обязательны**.

**Документ для пошагового контроля** (шаги, комментарии, галочки «готово»): [docs/STAGE2_SPACEWEB_PUBLIC.md](../docs/STAGE2_SPACEWEB_PUBLIC.md).

**Критерий готовности для руководителя:** открывается HTTPS-адрес API, проходит проверка «здоровья», один реальный файл успешно обрабатывается и скачивается — без настройки Mini App.

## Этап 3: улучшение UX веб-формы `/ui`

**Задача этапа:** сделать путь пользователя «**выбрал файл → обработал → скачал**» максимально понятным и предсказуемым, без изменения серверной логики.

**Что улучшаем для пользователя:**

- Явно показываем состояние процесса: **обработка / успех / ошибка**.
- Ошибки отображаем простым языком (вместо «технических» формулировок).
- В результате показываем ключевые показатели: время обработки и предупреждения валидации.
- Упрощаем действия после обработки: превью результата, скачивание, копирование ссылки.
- Добавляем защиту от случайных действий: запрет повторной отправки во время обработки и кнопка повторной попытки.

**Ожидаемый эффект для бизнеса:**

- Меньше «брошенных» попыток на шаге обработки.
- Меньше обращений в поддержку из-за непонятных ошибок.
- Выше доля успешных сценариев «загрузил и сразу получил готовый файл».

**Релизный подход:**

- Небольшие итерации с ручной проверкой после каждого шага.
- Минимальный риск: изменения только в интерфейсе, без перестройки инфраструктуры.
- Простой rollback: возврат предыдущей версии UI-файла.
