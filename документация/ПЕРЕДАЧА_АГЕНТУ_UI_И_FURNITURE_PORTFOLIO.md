## 1. Резюме по диалогу

### Продукт и репозиторий

- Репозиторий **Image Processing Gpt Agent** (`kamar009/image-processing-gpt-agent`): FastAPI, `POST /process-image`, UI [`static/index.html`](../static/index.html) (маршрут `/ui` → редирект на статику).
- Прод: в т.ч. `https://api.kimoffice9-18.ru`; Vision в проде — **Sber GigaChat-2-Max**.

### Сделано в коде (UI и API, до `furniture_portfolio`)

- **Карточный блок результата** в `/ui`: параметры, размер изображения/файла, техстрока (vision, всего, request_id, **HTTP в конце**).
- Режим **«Без модели»**: `vision_provider=fallback` — только программная обработка; поддержка в [`main.py`](../main.py) и селекте в `static/index.html`.
- Диагностика Vision fallback: поля **`vision_fallback_code`**, **`vision_fallback_message`** в ответе API; в т.ч. **`sber_oauth_401`** при ошибке OAuth Sber.
- Подсказки по полю **«Тип изображения»** (краткие пояснения + `aria-describedby`).
- **Деплой / SSH:** пользователь на VPS для `git pull` в `/opt/app` — **`deploy`**; локальный ключ **`~/.ssh/sweb_deploy_github2`**. Зафиксировано: [`.cursor/rules/sweb-ssh-deploy.mdc`](../.cursor/rules/sweb-ssh-deploy.mdc), [docs/SWEB_RUNBOOK.md](../docs/SWEB_RUNBOOK.md) §2b.

### Локальная диагностика Sber

- При **401** на `.../oauth` проверять **`SBER_VISION_AUTH_KEY`** / **`SBER_SCOPE`** в `.env`; тело ошибки может содержать `credentials doesn't match db data`.

### Новый продуктовый сценарий (согласовано текстом)

- Тип API **`furniture_portfolio`**: фото мебели после монтажа → портфолио (сайт + соцсети).
- Поля: **`furniture_scene`** (7 подтипов), **`output_target`** (site, banner, social_vk, social_telegram, social_max), **`enhanced`** (усиленная **программная** обработка v1 по флагу в запросе; без внешнего retouch).
- Люди в кадре: **предупреждение** без обязательного отказа, если `enhanced` выключен.

### Шаг A1 (документация)

- Выполнен: файл [docs/FURNITURE_PORTFOLIO_API.md](../docs/FURNITURE_PORTFOLIO_API.md), ссылка из [docs/INTEGRATION.md](../docs/INTEGRATION.md).

### Сделано в коде: тип `furniture_portfolio` (апрель 2026)

Сейчас новый режим доступен **через API** и через **веб-форму `/ui`**: при выборе типа «Мебель (портфолио)» показываются поля помещения, формата публикации и **галочка усиленной обработки** (`enhanced`); предупреждения API выводятся в карточке результата. Усиление включается **только** галочкой / полем `enhanced` в запросе.

Кратко для непрограммиста:

- Менеджер (или интегратор) отправляет фото, указывает тип **`furniture_portfolio`**, **сцену** (кабинет, переговорная и т.д.) и **куда картинка** (сайт, баннер, ВК, Telegram, MAX). Сервис подбирает **нужный размер** под площадку и прогоняет обработку.
- Слишком маленькое фото **отклоняется** с понятным текстом (длинная сторона после поворота по телефону должна быть не меньше порога из спеки).
- Нейросеть для этого режима получает **отдельные инструкции**: не «выдумывать» интерьер, не менять цвет мебели и форму модулей; если в кадре **видны люди**, в ответе будет **предупреждение**, но обработка может завершиться успешно (если файл прошёл проверки качества).
- Режим **«усиленная обработка»** (`enhanced`): **программный** пайплайн v1 (сильнее шум/резкость и т.д.; см. `operations`). Включается **флагом в запросе** (форма или API). **Удаление людей и внешняя ретушь** — не входят в v1; отдельный этап (**D+**) при необходимости.

## 3. Для руководителя (простыми словами)

### Зачем этот документ

Он связывает **продуктовую договорённость** (фото мебели на объекте → готовые картинки под сайт и соцсети) с **технической реализацией** и **планом на следующие шаги**, без погружения в код.

### Что уже даёт бизнесу сегодня (backend)

- Один и тот же сервис, что крутится у вас на проде с Vision (в т.ч. Sber), научился **новому типу заказа**: не просто «товар» или «интерьер», а именно **портфолио установленной мебели** с выбором формата вывода.
- **Ошибки в параметрах** (не та сцена, не тот формат, слишком маленький исходник) приходят **сразу и человеческим текстом**, а не «молчаливым сбоем».
- Если нейросеть считает, что **в кадре есть люди**, клиент получит **предупреждение в ответе** — чтобы не выложить кадр не глядя. Это не замена редактору, а страховка.
- Режим **усиленной программной обработки** (**`enhanced`**) включается **запросом** (галочка в `/ui` или поле формы в API): более сильная «локальная» доработка кадра **без** обещания удалить людей или сделать глянцевую ретушь.

### Чего ещё нет (и это нормально для этапа)

- **Программная** «усиленная обработка» в **`/ui`** и API есть (`enhanced`); это **не** внешняя ретушь.
- **Усиленная ретушь** (убрать людей, агрессивно почистить отражения через внешний сервис) — **не обещается** в текущей v1; при необходимости — отдельный этап согласования (**D1+**).

### Риски (честно)

- Без внешнего retouch (D1+) качество «как в глянце» **не гарантируется**; текущий режим — **аккуратная подготовка к публикации** и опциональное программное усиление v1, с соблюдением правил «не врать про мебель».
- Любой новый функционал на проде нужно **прогнать на ваших реальных фото** и при необходимости подкрутить пороги или тексты предупреждений.

### Как проверить, что всё живо (без программиста)

Попросить разработчика: один тестовый запрос к API с типом `furniture_portfolio`, корректными полями и нормальным по размеру фото; в ответе должны быть **размеры под выбранную площадку** и при необходимости **предупреждения** в списке warnings.

### Откат

Если что-то пойдёт не так после выката, технически можно **откатить коммит** с фичей или временно **не показывать** новый режим в клиентских формах — старые типы (`product`, `category`, `banner`, `portfolio_interior`) от этого **не ломались**.

*По запросу можно расшифровать любой шаг таблицы в §2: зачем он нужен, что проверить руками, что будет при откате.*
